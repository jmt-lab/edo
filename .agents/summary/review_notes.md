# Review Notes

Generated by the `codebase-summary` SOP. These observations track inconsistencies between documentation and code, as well as areas where the generated documentation could not be fully grounded.

## Documentation / Code Inconsistencies

### 1. Configuration language: docs say Starlark, code uses TOML

- `README.md` > "Starlark Configuration": _"Familiar, deterministic configuration language..."_
- `README.md` > "Configuration Reference": _"Edo uses Starlark for build configuration"_
- `docs/design.md` § 3.1 diagram references _"Starlark Engine"_ and _".edo build files"_.
- **Actual code**: `crates/edo-core/src/context/schema.rs` defines `Schema::V1` deserializing from TOML. Every `examples/**/edo.toml` declares `schema-version = "1"`. There is no `starlark` crate in `Cargo.toml` or `Cargo.lock`.
- **Recommendation**: update the public docs, or introduce Starlark support — pick one. AI agents will repeatedly be misled otherwise.

### 2. `docs/design.md` references a "Build Engine" type

- The design doc draws a boxed "Build Engine" component coordinating managers per domain.
- **Actual code**: responsibilities are split between `Context` (`edo-core/src/context/mod.rs`) and `Scheduler` (`edo-core/src/scheduler/mod.rs`). No struct named `Engine`/`BuildEngine` exists.
- **Recommendation**: rename in docs, or refactor to match docs.

### 3. README "Quick Start", "Configuration Reference", "Plugin Development" are TODO

- All three sections contain literal `_TODO_` / `__TODO__` placeholders.
- The only first-class introductory material for users is `examples/README.md` plus the two example projects, one of which is flagged as broken (`hello_oci`).

### 4. `hello_oci` example is documented as broken

- `examples/README.md`: _"this example is currently broken due to a bug in the container environment where the gcc image environment is not being fully loaded into podman correctly."_
- Leaving this in the repo as the canonical container example risks agents copying patterns that don't yet work.

### 5. `.gitignore` pattern for lock files is over-broad

- `.gitignore` contains `**.lock.json`. This would exclude `edo.lock.json` from version control — intentional if lock files are expected to be regenerated, but it also matches any file ending in `.lock.json` anywhere. Worth confirming intent.
- `examples/edo.lock.json` is present in the working tree, implying it is tracked despite the pattern (or was added before the pattern existed).

## Completeness Gaps

### A. `docs/components/` vs implementation drift

Per-component design docs (`docs/components/{storage,source,environment,transform,plugin}.md`) were not re-read in full during this pass. They were written alongside the Starlark-era design doc; agents should treat them as aspirational and confirm against trait signatures in `edo-core` before following them.

### B. `edo-wit` is not a Cargo crate

The `edo-wit` directory contains only `.wit` files and no `Cargo.toml`. It is neither a workspace member nor exclude'd. This currently works because both the host (`bindgen!`) and the SDK (`wit-bindgen`) resolve WIT files by relative path, but it is an undocumented convention that could bite future contributors.

### C. CI / test commands unknown

No `.github/workflows/`, no `Makefile`, no `justfile`, no `rust-toolchain.toml`, no obvious test harness notes. Standard `cargo build`, `cargo test`, `cargo clippy`, `cargo deny check` are presumed but unverified against a hypothetical CI matrix.

### D. Wasm plugin authoring has no end-to-end example

`edo-plugin-sdk` exists with `bindings.rs` + `Stub`, but there is no example `.wasm` plugin or integration test in the repository, and no documentation explaining which target (`wasm32-wasip2` for component model) to compile to.

### E. ScriptTransform variable substitution not documented

Examples use `{{install-root}}` and `{{build-root}}`. Whether other variables exist (e.g. `{{source-root}}`, `--arg`-supplied values exposed as variables, env vars) is not discoverable from a README — agents must read `crates/plugins/edo-core-plugin/src/transform/script.rs` to know.

### F. Storage backend addresses are implicit

The reserved addresses `//edo-local-cache`, `//edo-source-cache/<name>`, `//edo-build-cache`, `//edo-output-cache` are only visible in source comments in `crates/edo-core/src/storage/mod.rs`. They are not mentioned in public docs but are necessary to configure remote caches.

## Language-Support Limitations (of this pass)

This summary was produced entirely from Rust sources, TOML configs, WIT definitions, and Markdown docs. No other languages are in-tree, so there are no gaps from language limitations — with the caveat that guest-side plugin authoring could target **any language that compiles to a WebAssembly Component**, and such integrations are not represented in this repo.

## Recommendations

1. Reconcile Starlark-vs-TOML narrative across README and design docs (high priority — biggest agent-facing landmine).
2. Replace README TODOs with at minimum a pointer to the examples and to `edo-core-plugin` for "which transform kinds exist".
3. Either fix or remove the `hello_oci` example, or add a prominent banner in its `edo.toml`.
4. Document reserved storage addresses, Handlebars variables, and the `--arg` flow in one place.
5. Publish a minimal wasm plugin example under `examples/` or a new `plugins/` directory so `edo-plugin-sdk` has a real user.
6. Consider adding `crates/edo-wit` as a workspace member with a `Cargo.toml` containing a `build.rs` that validates WIT on build.
